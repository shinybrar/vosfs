# uv workspaces and Release Please: primary-source research

Researched: 2026-07-17

This note verifies the repository's two-package uv workspace and manifest-mode
Release Please configuration. It distinguishes tool guarantees from local
policy and records only facts supported by uv, Release Please, Ruff, pytest,
and ty documentation or source, plus the repository files that implement the
current design.

## Verified constraints

| Area | Primary-source fact | Consequence |
| --- | --- | --- |
| Workspace model | Every workspace package has its own `pyproject.toml`; `[tool.uv.workspace]` at the root declares required `members` globs and optional `exclude` globs. The workspace root is itself a workspace member, and the workspace shares one `uv.lock`. ([uv workspaces](https://docs.astral.sh/uv/concepts/projects/workspaces/)) | “Same `pyproject.toml`” should mean a shared workspace root, not shared package metadata. In this repository, `vosfs` is the root member whose distribution is under `src/vosfs`; `src/fsspec-cli` is the explicitly listed child member. It is inaccurate to call `vosfs` a non-member root project. ([root configuration](../../pyproject.toml), [member configuration](../../src/fsspec-cli/pyproject.toml)) |
| Members and virtual projects | Every included member directory must contain a `pyproject.toml`. A project can be made non-package/virtual with `tool.uv.package = false`; virtual projects contribute dependencies but are not built or installed. Workspace dependencies can also set `package = false`. ([workspace membership](https://docs.astral.sh/uv/concepts/projects/workspaces/), [uv `package` setting](https://docs.astral.sh/uv/reference/settings/#package), [virtual dependencies](https://docs.astral.sh/uv/concepts/projects/dependencies/#virtual-dependencies)) | Neither current project is virtual: both are independently buildable distributions with `[project]` and `[build-system]`. A virtual root would be a different architecture, not a prerequisite for a multi-package repository. ([root configuration](../../pyproject.toml), [member configuration](../../src/fsspec-cli/pyproject.toml)) |
| Package metadata | Workspaces manage packages together but do not combine their package metadata. Published dependencies belong in each package's `project.dependencies`; local development dependencies belong in that project's `[dependency-groups]`. ([uv workspaces](https://docs.astral.sh/uv/concepts/projects/workspaces/), [dependency fields](https://docs.astral.sh/uv/concepts/projects/dependencies/)) | Keeping runtime metadata and package-specific test requirements in each package is correct. Root `dev` is a sensible owner for repository-wide hooks, docs, lint, and type tools; `fsspec-cli`'s local `dev` group can retain only requirements needed when targeting that member. ([root groups](../../pyproject.toml), [member group](../../src/fsspec-cli/pyproject.toml)) |
| Dependency groups | `dev` is a default group for `uv run` and `uv sync`; other groups require selection unless added to `tool.uv.default-groups`. uv resolves all groups together, and group selection flags can include or exclude groups. ([dependency groups](https://docs.astral.sh/uv/concepts/projects/dependencies/#dependency-groups), [sync groups](https://docs.astral.sh/uv/concepts/projects/sync/#syncing-development-dependencies)) | The root `live` group is correctly opt-in. Shared tools need not be duplicated into the child `dev` group, provided commands that need both packages install all workspace members. ([root groups](../../pyproject.toml)) |
| Workspace sources | A dependency on another member must appear in a dependency field and have a corresponding `{ workspace = true }` source. Workspace members are editable. Root `tool.uv.sources` entries apply to members unless a member overrides the same dependency. Sources are uv-only and are ignored by other build tooling. ([workspace sources](https://docs.astral.sh/uv/concepts/projects/workspaces/#workspace-sources), [dependency sources](https://docs.astral.sh/uv/concepts/projects/dependencies/#dependency-sources)) | `fsspec-cli`'s development-only `vosfs` dependency plus `vosfs = { workspace = true }` is valid and does not create a published runtime dependency. ([member configuration](../../src/fsspec-cli/pyproject.toml)) |
| Lock ownership | `uv lock` operates on the complete workspace and the root owns the single universal `uv.lock`. The lock is generated by uv, should be version-controlled, and should not be edited manually. ([uv workspaces](https://docs.astral.sh/uv/concepts/projects/workspaces/), [lockfile](https://docs.astral.sh/uv/concepts/projects/layout/#the-lockfile)) | The repository's one root lock is correct. Both local package versions are represented in it: `fsspec-cli` as `0.1.1` and `vosfs` as `0.4.0`. ([manifest](../../.release-please-manifest.json), [`uv.lock`](../../uv.lock)) |
| Locked operation | `--locked` checks freshness and errors rather than updating a stale lock; `uv lock --check` is the explicit freshness check. `--frozen` uses the lock without checking project metadata. ([locking and syncing](https://docs.astral.sh/uv/concepts/projects/sync/)) | CI's `uv lock --check` and locked hook invocations protect against drift. Release-generated version changes must update both the relevant `pyproject.toml` and its local package entry in `uv.lock`. ([CI](../../.github/workflows/ci.yml), [hook configuration](../../.pre-commit-config.yaml)) |
| Sync target | `uv run` and `uv sync` target the root member by default. `--package` targets one member; `--all-packages` installs all workspace members and applies requested extras/groups to all of them. ([uv workspaces](https://docs.astral.sh/uv/concepts/projects/workspaces/), [uv CLI](https://docs.astral.sh/uv/reference/cli/)) | Plain `uv sync --locked` omits `fsspec-cli`'s dependencies, including `typer`. The repository already records this pitfall in `AGENTS.md`, and CI compensates with `uv run --all-packages`; `CONTRIBUTING.md` still gives the conflicting plain-sync setup command. ([agent instructions](../../AGENTS.md), [contributor setup](../../CONTRIBUTING.md), [CI](../../.github/workflows/ci.yml)) |
| Isolation limit | A shared Python environment cannot prove that one member imports only its declared dependencies. uv explicitly calls out this workspace limitation. ([when to use workspaces](https://docs.astral.sh/uv/concepts/projects/workspaces/#when-not-to-use-workspaces)) | The installed-wheel CI job is valuable: it tests `fsspec-cli` outside the workspace and catches undeclared dependencies that the shared `.venv` could hide. ([CI wheel job](../../.github/workflows/ci.yml), [ADR 0001](../adr/0001-release-fsspec-cli-independently.md)) |
| Build backend | `uv_build` defaults to one module at `src/<normalized_project_name>/__init__.py`; an upper bound is recommended because `uv_build` follows uv's versioning policy. It currently builds pure-Python packages. ([uv build backend](https://docs.astral.sh/uv/concepts/build-backend/)) | Both `vosfs` and `fsspec-cli` use the expected `src` layout and the documented bounded backend declaration. `fsspec-cli`'s `source-include` is package-local and intentionally adds its changelog/tests to the sdist. ([root build config](../../pyproject.toml), [member build config](../../src/fsspec-cli/pyproject.toml)) |
| Per-member build/publish | `uv build --package <name>` builds one package in the current workspace. uv recommends `uv build --no-sources` before publishing so a package is tested without uv-only source overrides. ([building and publishing](https://docs.astral.sh/uv/guides/package/)) | The CI and release workflows correctly select each package, but neither release build uses `--no-sources`; adding it would make the distribution gate closer to non-uv consumers. ([CI](../../.github/workflows/ci.yml), [vosfs publish](../../.github/workflows/publish.yml), [`fsspec-cli` publish](../../.github/workflows/fsspec-cli-publish.yml)) |
| Ruff configuration | Ruff uses the closest configuration containing `[tool.ruff]`, does not implicitly merge parent and child configurations, and supports explicit `extend`. A `pyproject.toml` without `[tool.ruff]` is ignored during discovery. ([Ruff configuration discovery](https://docs.astral.sh/ruff/configuration/#config-file-discovery)) | The member's `pyproject.toml` does not shadow the root Ruff policy, so the root policy applies to both trees. If member-specific Ruff settings are added later, they must explicitly `extend` the root or they will replace it. ([root Ruff config](../../pyproject.toml), [member configuration](../../src/fsspec-cli/pyproject.toml)) |
| pytest configuration | pytest selects one configuration while determining `rootdir`; candidate configuration files are not merged, and the first match wins. Native `[tool.pytest]` is supported in pytest 9. ([pytest configuration](https://docs.pytest.org/en/stable/reference/customize.html)) | The repository intentionally has two independent pytest policies: the root configuration controls `vosfs`, while the member-local table controls member tests. uv does not provide pytest inheritance between them. ([root pytest config](../../pyproject.toml), [member pytest config](../../src/fsspec-cli/pyproject.toml)) |
| ty configuration | ty searches the current directory and nearest parents for `ty.toml` or a `pyproject.toml` containing `[tool.ty]`; a `pyproject.toml` without that table is ignored. ([ty configuration](https://docs.astral.sh/ty/configuration/)) | The child metadata does not block discovery of the root ty configuration. The root command can therefore type-check both source trees, but its environment must include both packages' dependencies. ([root ty config](../../pyproject.toml), [hook command](../../.pre-commit-config.yaml)) |
| Release Please manifest mode | Manifest mode is designed for monorepos: package paths are keys in `packages`, released versions are keys in `.release-please-manifest.json`, and one config/manifest pair can manage multiple artifacts. `.` is the special root path. ([manifest releaser](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md)) | One shared config and manifest is the correct model for `.` and `src/fsspec-cli`; separate config files are neither required nor consistent with the accepted ADR. ([release config](../../release-please-config.json), [manifest](../../.release-please-manifest.json), [ADR 0001](../adr/0001-release-fsspec-cli-independently.md)) |
| Separate release PRs | `separate-pull-requests: true` opens one release PR per component instead of one combined manifest PR. ([manifest options](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md)) | The top-level setting implements the ADR's independent schedules. The action is correctly invoked once with the shared files. ([release config](../../release-please-config.json), [release workflow](../../.github/workflows/release.yml)) |
| Tags and action outputs | By default, tags contain the component and `v`; `include-component-in-tag: false` produces `v<version>`. For non-root paths, action outputs are prefixed with `<path>--`; root outputs are unprefixed. ([tag lookup](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md#subsequent-versions), [action outputs](https://github.com/googleapis/release-please-action#action-outputs)) | The configured and workflow-validated tags are coherent: `vosfs` gets `vX.Y.Z`; `fsspec-cli` gets `fsspec-cli-vX.Y.Z`. The workflow's `src/fsspec-cli--...` output names are correct. ([release config](../../release-please-config.json), [release workflow](../../.github/workflows/release.yml), [publish workflows](../../.github/workflows/publish.yml)) |
| Draft releases | `draft` creates a draft GitHub release. Release Please documents that draft releases may not create a tag until publication; `force-tag-creation` prevents subsequent release discovery from missing the previous release. ([schema](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json), [manifest options](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md)) | Root `vosfs` correctly combines `draft: true` with `force-tag-creation: true`. `fsspec-cli` has `draft: true` but inherits the false default for force-tag creation, even though its downstream workflow expects the exact tag immediately. ([release config](../../release-please-config.json), [`fsspec-cli` publish](../../.github/workflows/fsspec-cli-publish.yml)) |
| Python strategy | The Python strategy detects a package-relative `pyproject.toml` and adds a `PyProjectToml` version update automatically. ([Python strategy source](https://github.com/googleapis/release-please/blob/v17.6.1/src/strategies/python.ts)) | The member's string `extra-files` entry for `pyproject.toml` is redundant. As a generic TOML extra it targets `$.version`, while PEP 621 stores the value at `project.version`; the strategy's dedicated updater is the update that matters. ([release config](../../release-please-config.json), [member metadata](../../src/fsspec-cli/pyproject.toml), [base strategy source](https://github.com/googleapis/release-please/blob/v17.6.1/src/strategies/base.ts#L477-L503)) |
| TOML extra files | A TOML extra file requires `type`, `path`, and `jsonpath`. The GenericToml updater evaluates JSONPath against a tagged TOML parse and then replaces the selected original scalar. A leading `/` makes an extra-file path repository-root-relative instead of package-relative. ([customizing TOML](https://github.com/googleapis/release-please/blob/v17.6.1/docs/customizing.md#updating-arbitrary-toml-files), [GenericToml source](https://github.com/googleapis/release-please/blob/v17.6.1/src/updaters/generic-toml.ts), [path source](https://github.com/googleapis/release-please/blob/v17.6.1/src/strategies/base.ts#L753-L771)) | Both lock JSONPaths are correct for Release Please's tagged parse: `$.package[?(@.name.value=="…")].version`. The root entry uses `uv.lock`; the child correctly uses `/uv.lock` to escape package-relative resolution. A local execution of the pinned 17.6.1 updater changed only line 2384 for `vosfs` and only line 681 for `fsspec-cli`. ([release config](../../release-please-config.json), [`uv.lock`](../../uv.lock)) |
| Schema validation | The v17.6.1 schema permits manifest-level defaults and per-path package options and rejects unknown top-level properties, but the per-package option object does not set `additionalProperties: false`. ([v17.6.1 schema](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json)) | `release-please-config.json` passes the published schema. That does not make the nested `"path": "src/fsspec-cli"` meaningful: the package-map key already supplies the path and `path` is not a documented `ReleaserConfigOptions` property. It should be removed as ignored/redundant configuration. ([release config](../../release-please-config.json), [manifest package keys](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md#configfile)) |
| Excluded commits | A package `exclude-paths` entry excludes a commit only when all files in that commit belong to excluded paths. A configured package path already scopes a subdirectory release to commits touching that path. ([schema description](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json), [subdirectory behavior](https://github.com/googleapis/release-please/blob/v17.6.1/docs/customizing.md#subdirectories-paths-in-a-repository)) | Root exclusions for `src/fsspec-cli/**`, its workflows/design documents, and shared `uv.lock` prevent component-only work from causing a root release. The entries for `.fsspec-cli-release-please-manifest.json` and `fsspec-cli-release-please-config.json` are stale because those files do not exist. The child's `exclude-paths: ["src/vosfs"]` is redundant under its `src/fsspec-cli` package scope. ([release config](../../release-please-config.json), [release policy](../agents/release.md)) |
| Pre-1.0 bump policy | `bump-minor-pre-major` makes breaking changes bump minor below 1.0; `bump-patch-for-minor-pre-major` makes features bump patch below 1.0. Both default false. ([schema](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json), [manifest options](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md)) | The policies are materially asymmetric: `vosfs` uses ordinary SemVer bumping, while `fsspec-cli` compresses breaking changes to minor and features to patch. No rationale for that difference appears in ADR 0001 or the release policy, so it is policy rather than a verified requirement. ([release config](../../release-please-config.json), [ADR 0001](../adr/0001-release-fsspec-cli-independently.md), [release policy](../agents/release.md)) |

## Current repository assessment

The package topology is sound. The root `pyproject.toml` is simultaneously the
metadata file for the `vosfs` distribution and the uv workspace root; that root
project is a workspace member. `src/fsspec-cli/pyproject.toml` is the second
package's independent metadata file. Both packages use bounded `uv_build`
requirements and conventional `src` layouts, and both are locked in one
root-owned `uv.lock`. These observations follow the documented workspace model,
not an exception to it. ([uv workspaces](https://docs.astral.sh/uv/concepts/projects/workspaces/),
[root metadata](../../pyproject.toml),
[member metadata](../../src/fsspec-cli/pyproject.toml),
[`uv.lock`](../../uv.lock))

Dependency ownership is also mostly appropriate. Runtime requirements are
package-local, repository-wide development tooling is rooted, the live
credential dependency is opt-in, and the member's development relationship to
`vosfs` is expressed as an editable workspace source rather than a published
runtime requirement. ([uv dependency fields](https://docs.astral.sh/uv/concepts/projects/dependencies/),
[root groups](../../pyproject.toml),
[member dependencies](../../src/fsspec-cli/pyproject.toml))

The concrete uv inconsistency is documentation: `AGENTS.md` correctly requires
`uv sync --locked --all-packages`, while the contributor bootstrap still says
`uv sync --locked`. This is observable, not theoretical: a dry run of the plain
sync would remove the member-only `typer` stack, whereas the all-package dry run
retains it. The nested locked commands in the local hooks then rely on that
all-package environment to type-check `fsspec_cli`. ([uv CLI
`--all-packages`](https://docs.astral.sh/uv/reference/cli/),
[agent instructions](../../AGENTS.md),
[contributor setup](../../CONTRIBUTING.md),
[hooks](../../.pre-commit-config.yaml))

The Release Please architecture is likewise correct: one manifest-mode action
owns two path keys, creates separate release PRs, emits the expected root and
path-prefixed outputs, and dispatches package-specific builds. The configuration
validated successfully against its declared v17.6.1 schema. ([manifest
releaser](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md),
[action inputs and outputs](https://github.com/googleapis/release-please-action),
[release workflow](../../.github/workflows/release.yml),
[schema](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json))

The `uv.lock` extra-file mechanism is verified for this exact lock structure and
Release Please version. The unusual `.name.value` filter is required by the
updater's tagged TOML parser; omitting `.value` would not select the intended
entry. The child `/uv.lock` leading slash is also intentional because Release
Please otherwise resolves extras relative to `src/fsspec-cli`. ([GenericToml
source](https://github.com/googleapis/release-please/blob/v17.6.1/src/updaters/generic-toml.ts),
[TOML edit source](https://github.com/googleapis/release-please/blob/v17.6.1/src/util/toml-edit.ts),
[base path source](https://github.com/googleapis/release-please/blob/v17.6.1/src/strategies/base.ts#L753-L771),
[release config](../../release-please-config.json))

## Recommended decisions

1. **Use all-package setup as the canonical developer bootstrap (highest
   impact).** Change `CONTRIBUTING.md` to
   `uv sync --locked --all-packages`. Keep `uv lock --check` in CI and keep
   targeted `uv run --package fsspec-cli ...` commands for package-specific
   tests. This aligns the human setup with the already-correct agent and CI
   behavior. ([uv workspace command behavior](https://docs.astral.sh/uv/concepts/projects/workspaces/),
   [uv CLI](https://docs.astral.sh/uv/reference/cli/),
   [local conflict](../../CONTRIBUTING.md))

2. **Set `force-tag-creation: true` for `fsspec-cli` (high impact).** Its
   releases are drafts and the publication workflow immediately checks out an
   exact `fsspec-cli-vX.Y.Z` tag. Release Please explicitly documents the
   previous-release discovery hazard when draft releases defer tag creation.
   ([manifest option](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md),
   [schema](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json),
   [local workflow](../../.github/workflows/fsspec-cli-publish.yml))

3. **Retain one config, one manifest, and both targeted lock updates (high
   confidence, no architecture change).** This is the native manifest-monorepo
   model and matches ADR 0001. Do not restore the obsolete separate
   `fsspec-cli` config/manifest files. ([manifest
   releaser](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md),
   [ADR](../adr/0001-release-fsspec-cli-independently.md))

4. **Remove semantically redundant release settings (medium impact).** Delete
   the child `"path"` property, string `extra-files: "pyproject.toml"`,
   `changelog-path: "CHANGELOG.md"`, and `prerelease: false`; delete root
   `versioning: "default"` if explicit defaults are not a local documentation
   convention. The package map, Python strategy, and documented defaults
   already provide those values. Keep the object-form `/uv.lock` extra.
   ([manifest config model](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md),
   [Python strategy](https://github.com/googleapis/release-please/blob/v17.6.1/src/strategies/python.ts),
   [schema defaults](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json))

5. **Prune stale and ineffective exclusions (medium impact).** Remove the two
   nonexistent separate-release filenames from the root list and remove the
   child `src/vosfs` exclusion. Preserve exclusions that prevent
   `fsspec-cli`-only changes and lock-only version updates from entering root
   commit analysis. ([exclude semantics](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json),
   [local config](../../release-please-config.json))

6. **Make pre-1.0 versioning policy explicit (medium governance impact).**
   Either document why `fsspec-cli` treats features as patches and breaking
   changes as minors while `vosfs` does not, or align the booleans. Do not
   normalize them merely for symmetry: changing these flags changes released
   versions. ([option semantics](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json),
   [current asymmetry](../../release-please-config.json))

7. **Add source-independent build gates (lower immediate impact).** Use
   `uv build --no-sources --package vosfs` and
   `uv build --no-sources --package fsspec-cli` in the release or pre-release
   build checks. Keep the installed-wheel test because `--no-sources` does not
   replace import isolation. ([uv publishing
   guidance](https://docs.astral.sh/uv/guides/package/),
   [workspace isolation limit](https://docs.astral.sh/uv/concepts/projects/workspaces/#when-not-to-use-workspaces),
   [current CI](../../.github/workflows/ci.yml))

8. **Preserve deliberate tool discovery boundaries (lower immediate impact).**
   Keep one root Ruff and ty policy while the member has no overrides; keep
   pytest package-local. If a member later adds Ruff configuration, use
   `extend = "../../pyproject.toml"`; do not assume uv workspace inheritance.
   ([Ruff discovery](https://docs.astral.sh/ruff/configuration/#config-file-discovery),
   [pytest discovery](https://docs.pytest.org/en/stable/reference/customize.html),
   [ty discovery](https://docs.astral.sh/ty/configuration/))

## Deliberately unresolved

- This research does not choose whether pre-1.0 releases should follow strict
  SemVer-major behavior or compressed minor/patch behavior. Release Please
  defines both mechanisms, but product compatibility policy owns the choice.
  ([Release Please schema](https://raw.githubusercontent.com/googleapis/release-please/v17.6.1/schemas/config.json))
- It does not decide whether repository-wide development dependencies should be
  split into nested `lint`, `test`, and `docs` groups. uv supports nested and
  selectable groups, but the current single root `dev` group is valid and the
  maintenance trade-off is local. ([dependency
  groups](https://docs.astral.sh/uv/concepts/projects/dependencies/#dependency-groups))
- It does not propose converting the root to a virtual workspace. That would
  require moving `vosfs` to another member and would change the accepted release
  path/tag model; no tool constraint requires it. ([uv virtual
  projects](https://docs.astral.sh/uv/reference/settings/#package),
  [ADR 0001](../adr/0001-release-fsspec-cli-independently.md))
- It does not decide whether GitHub draft releases should remain part of the
  publication safety model. If drafts remain, both package entries should force
  tag creation; if drafts are removed, the downstream validation/publish flow
  must be reconsidered together. ([manifest release
  options](https://github.com/googleapis/release-please/blob/v17.6.1/docs/manifest-releaser.md),
  [publish workflows](../../.github/workflows/publish.yml))
- Schema validation proves structural acceptance, not that Release Please will
  find commits or create releases against live GitHub history. No live release
  PR or tag was created as part of this read-only research.

## Concrete improvements for vosfs

1. Change contributor setup to `uv sync --locked --all-packages`.
2. Add `force-tag-creation: true` to the `src/fsspec-cli` release entry.
3. Keep the shared config/manifest and both verified `uv.lock` JSONPath updates.
4. Remove the child `path`, redundant `pyproject.toml` extra, redundant defaults,
   and stale separate-config exclusions.
5. Remove the child `exclude-paths: ["src/vosfs"]` entry while retaining root
   exclusions that isolate `fsspec-cli` commits.
6. Document or align the two packages' pre-1.0 bump policies.
7. Add `--no-sources` to package build gates before attaching release artifacts.
